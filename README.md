# Supply Chain Optimisation — End-to-End PoC Demo

## Scenario: FreshBox Australia

**FreshBox Australia** is a meal-kit and grocery-box delivery company based in Melbourne.
They pack and deliver ~15,000 food boxes per week across Victoria, NSW, and Queensland.
They carry ~200 SKUs (fresh produce, proteins, pantry items, packaging).

Their long-tenured warehouse manager, Dave, is retiring. Dave keeps most replenishment
logic in his head. The company needs to capture Dave's knowledge and build a system
that new staff can rely on.

This demo walks through every step of building that system — from raw data through
to an interactive decision-support dashboard.

---

## Dataset Options

The pipeline supports two data sources, switchable in `config/settings.py`:

### Option A — Synthetic data (default for new users, no download needed)

Generated by `src/00_generate_synthetic_data.py`. Mimics the Corporación Favorita
structure: 5 stores, 50 items, ~2.5 years of daily sales including perishable flags,
promotions, holidays, and store metadata. Uses a fixed random seed so results are
fully reproducible.

In `config/settings.py`:
```python
DATA_SOURCE = "synthetic"
```

### Option B — Real Kaggle Favorita data (125M rows, richer signal)

The Kaggle Corporación Favorita CSVs are included in the `kaggle_data_files/` folder
inside the project — no separate download required. The pipeline is already configured
to use them:

In `config/settings.py`:
```python
DATA_SOURCE = "kaggle"
KAGGLE_DATA_PATH = BASE_DIR / "kaggle_data_files"  # already set — do not change
```

The pipeline validates that `train.csv` exists and raises a clear error if not.

> **Results with Kaggle data:** fill_rate=98.4%, waste_rate=4.5%, sMAPE=34.2%, total_cost=$3.94M

## Quick Start

### 1. Environment Setup

```bash
# Python 3.10+ required
python -m venv .venv
.venv\Scripts\activate          # Windows
# source .venv/bin/activate     # macOS / Linux

pip install -r requirements.txt
```

### 2. Choose your data source

Edit `config/settings.py` and set `DATA_SOURCE`:

```python
DATA_SOURCE = "kaggle"      # Kaggle data in kaggle_data_files/ — already set
# DATA_SOURCE = "synthetic"  # fallback: no CSVs needed, generates data on the fly
```

### 3. Run the full pipeline

```bash
python run_pipeline.py
```

This runs **pytest first** (21 tests must pass), then steps 1–7 in order, then
**automatically launches the Streamlit dashboard** at http://localhost:8501.

#### Resuming from a specific step

If a step fails mid-run and you fix it, you can skip re-running earlier steps:

```bash
python run_pipeline.py --from-step 4    # restart from demand forecasting
python run_pipeline.py --from-step 6    # restart from inventory optimisation
```

#### Other flags

| Flag | Effect |
|------|--------|
| `--from-step N` | Skip steps 1 through N-1; resume at step N (1–7) |
| `--skip-tests` | Bypass pytest (not recommended for production) |
| `--no-dashboard` | Don't launch Streamlit at the end |

Flags can be combined: `python run_pipeline.py --from-step 6 --skip-tests`

### 4. Run individual steps

Each script can also be run independently:

```bash
python src/01_data_prep.py
python src/02_eda.py
python src/03_feature_engineering.py
python src/04_demand_forecasting.py
python src/05_business_rules.py
python src/06_inventory_optimisation.py
python src/07_evaluation.py
streamlit run src/08_dashboard.py
```

## Dashboard Guide

The dashboard (`src/08_dashboard.py`) is a Streamlit web app that shows the
optimisation results to warehouse operators — the people replacing Dave.

### What it shows

| Section | Description |
|---------|-------------|
| **KPI cards** | Fill rate, waste rate, total cost, stockout count at a glance |
| **Order plan table** | Item-by-item recommended order quantities for the selected store + week |
| **Forecast chart** | Actual demand vs q10/q50/q90 forecast bands, with confidence intervals |
| **Business rules log** | Which of Dave's rules fired, and what adjustments were made |
| **Cost breakdown** | Procurement vs waste cost vs stockout cost, as a chart |

### How to use it

1. **Select a store** from the sidebar dropdown
2. **Select a week** — the order plan for that week appears
3. **Review the order quantities** — green rows meet demand comfortably, amber rows are close to safety stock limits
4. **Check the rules log** — see if any seasonal/promotional adjustments were applied automatically
5. **Override if needed** — for any item you disagree with, you can adjust quantities directly. Overrides are logged for future model improvement.

The system is designed as *decision support*, not automation. A new operator
should be able to use it with no knowledge of Dave's rules — the rules are
encoded in the system and the reasoning is shown.

5. App is live at `https://your-app-name.streamlit.app` within ~2 minutes

**Access control (public vs private):**
- After deployment, go to **Settings → Sharing**
- Set to **Private** and add specific email addresses to restrict access — useful for client demos
- The free tier supports private apps at no cost

#### 2. Hugging Face Spaces (free, good for ML demos)

- Create a Space at https://huggingface.co/spaces with SDK = Streamlit
- Push the repo — Hugging Face builds and serves it automatically
- Free tier includes persistent storage, so you can store the processed parquets

#### 3. Railway / Render (free tier, more control)

- Both support Docker or Python apps with a `requirements.txt`
- Add a `Procfile`: `web: streamlit run src/08_dashboard.py --server.port $PORT --server.address 0.0.0.0`
- Connect your GitHub repo — auto-deploys on push

#### What about Cloudflare?

Cloudflare Pages/Workers are designed for static sites and JavaScript edge functions —
they cannot run a Python process. You would need to deploy the Python backend
(Streamlit) elsewhere and optionally put Cloudflare in front of it as a CDN/proxy.
For this use case, Streamlit Community Cloud or Hugging Face Spaces are simpler.

## Glossary of Key Terms

| Term | Meaning |
|------|---------|
| **SKU** | Stock Keeping Unit — a unique identifier for each product |
| **Quantile forecast** | Instead of a single "best guess", predict the 10th/50th/90th percentile of demand. The 90th percentile means "there's only a 10% chance demand exceeds this" |
| **Safety stock** | Extra inventory held as a buffer against demand uncertainty. Derived from the gap between q50 (median) and q90 (high scenario) forecasts |
| **MOQ** | Minimum Order Quantity — the smallest amount a supplier will sell |
| **Lead time** | Days between placing an order and receiving it |
| **Fill rate** | % of demand met from available stock (target: >98%) |
| **sMAPE** | Symmetric Mean Absolute Percentage Error — `2|A-F|/(|A|+|F|)`. More robust than MAPE for low-volume items because both actuals and forecasts appear in the denominator |
| **MIP** | Mixed-Integer Programming — optimisation where some variables must be whole numbers |
| **Conformal prediction** | A distribution-free method to construct statistically valid prediction intervals. We use the calibration set to widen q10/q90 until 80% empirical coverage is achieved |
| **q40 perishable demand** | For short-shelf-life items, we use the 40th percentile as the base order quantity (interpolated: `q10 + 0.75*(q50-q10)`) to reduce over-ordering and waste |
| **Intermittent demand** | Items with very low average weekly sales (< 20 units/week). Excluded from the top-N selection because LightGBM has insufficient signal to forecast them reliably |
| **Newsvendor model** | A classic single-period inventory model that balances overage cost (waste) vs underage cost (stockout) |
| **FEFO** | First Expired, First Out — for perishables, use the stock closest to expiry first |
| **Objective function** | The thing the optimiser is trying to minimise — here, total cost (procurement + waste + stockout) |
| **Constraint** | A rule the optimiser must obey — e.g. "order at least MOQ" or "don't exceed warehouse capacity" |

## Objective Function (Step 6)

The MIP solver minimises total supply chain cost across all items `i` in a store-week:

$$\min \sum_i \Big[ c_p \cdot q_i + w_i \cdot p_i \cdot \text{overstock}_i + c_s \cdot \text{understock}_i \Big]$$

Where:

| Symbol | Parameter | Value |
|--------|-----------|-------|
| $c_p$ | Procurement cost per unit | $4.00 |
| $c_s$ | Stockout cost per unit | $10.00 |
| $w_i$ | Waste cost per unit | $3.50 (non-perishable) / $8.75 (perishable, ×2.5 multiplier) |
| $p_i$ | Waste probability | 10% (non-perishable) / 40% (perishable) |
| $w_i \cdot p_i$ | **Effective waste cost** | **$0.35/unit** (non-perish.) / **$3.50/unit** (perish.) |

The cost ratios imply a **critical ratio** (the service level the solver targets):

$$CR = \frac{c_s}{c_s + w_i \cdot p_i} = \begin{cases} 96.6\% & \text{non-perishable} \\ 74.1\% & \text{perishable} \end{cases}$$

This means the solver leans heavily toward avoiding stockouts for non-perishables, and is more balanced for perishables (where waste cost is significant).

### Constraints

| Constraint | Purpose |
|-----------|---------|
| `order_qty[i] + on_hand[i] ≥ demand[i] + safety_stock[i]` | Meet expected demand plus buffer |
| `overstock[i] ≥ order_qty[i] + on_hand[i] − demand[i]` | Define overstock for cost calculation |
| `understock[i] ≥ demand[i] − order_qty[i] − on_hand[i]` | Define understock (stockout risk) |
| `order_qty[i] ≥ MOQ × use_supplier[i]` | Enforce minimum order quantity if ordering |
| `Σ_i (order_qty[i] + on_hand[i]) ≤ warehouse_capacity` | Respect warehouse capacity per store |
| `order_qty[i] ≥ 0`, integer | Whole units only |

Perishable items use a conservative base demand (≈ q40 instead of q50) to avoid over-ordering short-shelf-life products.
